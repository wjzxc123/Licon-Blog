## 一些疑问以及理解

### 现如今 AI 是什么？

现在大家日常接触到的 AI，大多是基于深度学习训练出来的模型，尤其是大语言模型（LLM）。

它们在大量数据里学习语言、知识和模式，然后在你输入一句话之后，根据上下文继续预测下一个 token。对外看起来像是在“理解”和“交流”，底层做的其实是高维空间里的概率计算。

这也是大模型很有意思的地方。预测下一个 token 这件事听起来很朴素，但语言、代码、图像描述、工具指令，本来就都可以表示成 token 序列。于是同一套机制，最后长出了写作、翻译、总结、编程、问答这些能力。

至于“意识”这件事，至少在今天，还没有可靠证据表明大模型已经拥有人类意义上的主观意识。它更像一个能力极强的模式学习系统。

### 为什么大模型通常要先训练，再推理，而不是边学边答？

大模型的工作方式可以拆成两层来看。

一层是推理时利用上下文继续回答。你和它对话越多，它越能顺着上下文往下说，这属于“在当前窗口里理解你”。另一层才是“真正学会了新知识”，那需要改动模型参数，通常要经过继续预训练、微调或者对齐训练。

现实里的大模型服务，基本都会把这两件事分开。因为如果每次对话都现场改权重，成本会很高，风险也很大，模型还容易把错误信息学进去，最后行为越来越不稳定。

所以今天更常见的路线是：参数相对固定，知识更新依靠再训练、微调、RAG（检索增强生成）和工具调用来完成。

### AI 在“创造”方面到底和人有什么区别？

我越来越觉得，AI 的“创造力”本质上是一种高强度的重组能力。

它能把见过的大量模式重新拼接、迁移、组合，生成新的表达、新的结构，甚至新的解决方案。写文案、想标题、改代码、做设计草图，它常常都能给出挺像样的结果。

人类的创造也不是凭空出现的。我们的经验、语言、文化、情绪、身体感受，本来就在塑造我们的想法。AI 和人的差别，不在于“一个会创造，一个不会”，而在于创造背后的驱动力不同。

人类会主动设定长期目标，会为偏好和价值负责，也会因为真实世界里的经历改变判断。AI 在这方面还差得很远。它很擅长在已有边界内生成精彩内容，但离“自己决定要成为什么，并长期为此行动”还有明显距离。

## AI 的基础路线

如果把时间线往前拨，在 Transformer 成为主角之前，深度学习世界里已经有很多重要角色了。

CNN 在图像任务里非常耀眼，特别擅长提取局部空间特征。RNN、LSTM、GRU 则长期活跃在文本、语音和时间序列建模里。直到今天，这些架构依然没有消失，只是各自回到了更适合自己的位置上。

真正把大语言模型推上快车道的，是 2017 年提出的 Transformer。它让模型更容易并行训练，也更擅长处理长距离依赖关系。模型能做得更大，训练速度能提上来，很多后来我们看到的突破，都是从这里开始积累出来的。

## 什么是 Transformer 架构

Transformer 最核心的地方，是 Self-Attention，也就是自注意力。

它的直觉很好理解：模型在处理当前 token 的时候，不是只盯着前后几个词，而是会动态参考整段序列里和它相关的部分。于是“这个词到底在说谁”“前面那个代词指的是什么”“这一句和上文真正相关的是哪一段”，都能被更自然地建模出来。

如果只用一句话概括 Transformer 的意义，我会写成这样：

> Transformer 让模型更会看上下文，也更适合在并行硬件上训练成超大规模。

今天很多大语言模型，比如 GPT 这一类，采用的是 decoder-only Transformer。它们很适合做自回归生成，也就是一个 token 一个 token 地往后接着写。

## 有了架构之后，怎么把它写成代码？

架构图画出来只是开始，真正落地还是工程问题。

理论上当然可以从底层矩阵运算开始自己写，但真要这么干，工作量会非常惊人。现实里的做法，通常是直接站在成熟框架和开源库的肩膀上：

- PyTorch
- TensorFlow
- JAX
- [Hugging Face Transformers](https://github.com/huggingface/transformers)

其中 `Transformers` 更像一个工程化模型库，里面不止有 Transformer 的实现，也有完整的模型加载、训练、推理和生态工具。

如果今天自己上手写模型代码，PyTorch 依然是最常见的起点。生态成熟，调试友好，资料足够多，上手和扩展都更顺手。

## 有了架构之后，如何真正训练出一个模型？

有了架构，离“训练出一个能用的模型”还差很多东西。

首先是数据。数量要够，质量要稳，覆盖面要广，还要做去重、清洗、过滤，很多场景里还要考虑授权和合规。然后是 tokenizer，它决定文本怎么被切成 token。再往后是训练目标、优化器、学习率、批大小、混合精度、并行策略、评测方式，还有整个训练过程的稳定性。

算力当然也绕不过去。大模型训练本质上是超大规模矩阵运算，所以 GPU、TPU 这类并行硬件会成为核心资源。现在最常见的训练生态依然围绕 NVIDIA 和 CUDA 展开，但它也不是唯一答案，TPU、AMD GPU 和其他加速器同样在很多地方落地。

参数规模很重要，但模型能力从来不是只由“参数大不大”决定的。数据质量、训练方法、上下文长度、后训练方式、推理策略，这些都会直接影响最终效果。

如果不是从零开始造模型，今天其实有非常多现成资源可以利用，比如 Hugging Face 上的 [datasets](https://huggingface.co/datasets)。

## 训练完之后会得到什么？

训练结束以后，通常会得到几类文件：

- 权重文件：例如 `.safetensors`、`.bin`
- 配置文件：例如 `config.json`
- Tokenizer 文件：例如 `tokenizer.json`、`vocab.json`、`merges.txt`
- 额外元数据：特殊 token 配置、生成参数、模型卡等

这些文件里最核心的是模型参数，也就是神经网络各层的矩阵权重。token 是输入输出的离散表示，真正负责计算和生成能力的，是模型内部这些参数。

不同模型的结构也不完全一样。像 GPT 这类主流大语言模型，通常采用 decoder-only 结构；像 T5 这类模型，则属于典型的 encoder-decoder。

如果算力不够，从头训练通常不是最划算的路线。更常见的做法是：

- 直接使用开源基座模型
- 做 LoRA / 全量微调
- 只做推理部署
- 在应用层加 RAG、工具调用和工作流

可选模型可以在 Hugging Face 的 [models](https://huggingface.co/models) 上找到。

## 有了模型之后怎么用？

模型真正跑起来，靠的是推理（inference）。

整个过程大体是这样：

1. 把输入文本切成 token
2. 模型根据上下文计算下一步最可能的 token 分布
3. 按照采样策略或贪心策略生成新 token
4. 再把 token 还原成文本

如果只是本地实验或者做原型，`PyTorch + Transformers` 就已经很好用了。

一旦到了线上服务、并发请求、吞吐优化这些层面，大家更常接触到的是这些推理框架：

- `vLLM`
- `TGI`
- `llama.cpp`
- `Ollama`

它们分别在部署方式、吞吐优化、硬件适配和易用性上做了很多工程化取舍，所以同样是“跑模型”，体验会差很多。

## 模型能对话了，就结束了吗？

还远远没有。

模型会回答问题，只代表它具备了基础生成能力。真正好用的 AI 应用，往往是在模型之外又包了一整层能力。

你会需要系统提示词来约束角色和输出格式，会需要 RAG 把外部知识接进来，会需要工具调用让模型查、算、搜、执行操作，还会需要工作流把这些环节串起来。再往前走一步，还得有评测、监控、回放和持续优化机制，不然应用一旦复杂起来，问题会非常难排查。

大众真正感受到对话式 AI 爆发的节点，是 ChatGPT 在 2022 年 11 月 30 日发布。到了 2023 年，整个行业才进入应用全面出圈、产品快速爆发的阶段。

## 从聊天到 AI 应用

写 AI 应用这件事，Prompt 很重要，但 Prompt 只是接口的一部分。

真正决定体验的，通常是“模型 + 数据 + 工具 + 工作流 + 评测”这几个部分怎么配合。

像 `LangChain` 这类框架，做的事情就是把模型调用、Prompt 模板、检索、记忆、工具调用这些能力封装起来，让开发者能更快地拼出一个应用原型。

如果想把流程编排得更清楚，比如：

- 先检索资料
- 再判断是否足够回答
- 不够就调用搜索或数据库
- 最后整理答案并输出固定格式

这类场景通常会用到 `LangGraph`、自定义工作流引擎，或者直接自己实现状态机。

很多人把“AI 应用开发”想成了“写 Prompt 工程”，但真正落地的时候，它更像在搭一套有模型参与决策的软件系统。

## Agent 是什么？

Agent 这个词这两年很火，也很容易被讲得很玄。

> Agent 是一个能在目标驱动下，结合模型、记忆、工具和执行流程，分步骤完成任务的系统。

它可以是单模型驱动，也可以是多 Agent 协作。关键不在数量，而在于它能不能围绕目标持续拆任务、做决策、调用工具、追踪状态，再把结果收回来。

Agent 之所以看起来比普通聊天更像“会做事”，通常是因为它多了这些能力：

- 任务分解
- 工具调用
- 外部记忆
- 状态跟踪
- 错误重试
- 结果验证

## openClaw 这类项目在做什么？

有一类 Agent 更进一步，它的目标不只是回答问题，而是直接去操作界面、调用软件、执行任务。

这类系统看起来像“AI 接管电脑”，背后做的事情其实很朴素：

- 读取屏幕或界面状态
- 理解当前任务目标
- 决定下一步操作
- 点击、输入、切换页面
- 观察结果并继续迭代

这类方向非常有想象空间，因为它把模型从“会说话”推进到了“会动手”。但它距离稳定、可靠、可放心托付，还有很长一段路。

一旦任务涉及付款、删除数据、交易、系统权限这些高风险动作，权限控制和人工兜底仍然是必须的。
